## 2. 技术选型

| 组件 | 方案 | 选型原因 |
|------|------|----------|
| 编程语言 | Python 3.8+ | 团队熟悉，生态丰富 |
| 检索引擎 | scikit-learn TfidfVectorizer | char_wb n-gram，中文自适应 |
| 关键词匹配 | BM25 | 词频饱和 + 文档长度归一化 + IDF 稀有词加权，零依赖 |
| 语义向量 | BGE-small-zh-v1.5 (HuggingFace) | 中文优化，CPU 可跑，24MB |
| 融合排序 | RRF (Reciprocal Rank Fusion) | 无需权重调参，自动平衡稀疏/稠密信号 |
| 分块策略 | 段落分块 + 滑动窗口 | 保留语义边界，长段落自动切分 |
| 分词策略 | 正则单字 + 英文整词 | 中文细粒度匹配，英文完整保留 |
| 序列化 | pickle | Python 原生，读写速度快 |
| Web 框架 | http.server (标准库) | MVP 场景够用 |
| 前端 | 内联 HTML + 原生 JS | 单文件分发，零构建步骤 |
| 网页抓取 | requests + BeautifulSoup4 | HTML 解析和 CSS 选择器支持 |
| 企微加解密 | pycryptodome | AES-256-CBC 加解密 |

### 3.3 search.py — 混合搜索引擎模块

**职责:** 统一的搜索模块，CLI 和 Web 服务共享。加载索引 → 分词 → TF-IDF + BM25 (+ Embedding) → RRF 融合 → 多层过滤 → 返回结果。

**分词策略 (`tokenize`):**

```
"开发环境 VS Code"  →  ["开", "发", "环", "境", "vs", "code"]
 ────────────────        ─────────────────────────────────
 中文: 单字切分           英文: 整词保留
（确保细粒度匹配）         （避免 "environment" 被拆散）
```

**BM25 算法:**

```
BM25(D, Q) = Σ IDF(qi) × f(qi,D) × (k1+1) / (f(qi,D) + k1 × (1-b + b×|D|/avgdl))

其中:
  IDF(qi) = log((N - df + 0.5) / (df + 0.5) + 1)
  f(qi, D) = 词 qi 在文档 D 中的频率
  |D| = 文档词数, avgdl = 平均文档长度
  k1 = 1.5 (词频饱和), b = 0.75 (长度归一化)
```

与 TF-IDF 的互补关系：

| 维度 | TF-IDF (char_wb) | BM25 |
|------|-----------------|------|
| 匹配粒度 | 2-4 字符 n-gram | 中文单字 + 英文整词 |
| 词频处理 | 无饱和 | k1 参数控制饱和 |
| 文档长度 | 无归一化 | b 参数控制归一化 |
| 稀有词 | 隐式（IDF 在矩阵中） | 显式 IDF 公式 |
| 优势 | 模糊匹配、部分重叠 | 精确匹配、区分度 |

**RRF 融合排序:**

```
RRF_score = Σ 1/(k + rank_i)    k = 60

排名用 Dense Ranking: 同分同名次，零分强制末位 (rank = n)
归一化: score / max_possible → 0~1 区间
```

**多层过滤链:**

```
查询
  │
  ├── ① MIN_TFIDF: max(raw_tfidf) < 0.05?  → 无结果
  ├── ② BM25 零检查: max(bm25) == 0?        → 无结果（查询词不存在于语料库）
  ├── ③ Dense ranking + 零分块末位惩罚
  ├── ④ RRF 融合 + 归一化
  ├── ⑤ TF-IDF=0 的块 → score 移除 tfidf 部分贡献
  └── ⑥ MIN_RELEVANCE: score < 0.60?       → 排除
```

每层的设计目的：

| 层 | 拦截场景 | 示例 |
|----|----------|------|
| ① | 无意义字符串、纯英文与中文文档不沾边 | `drmtest` → max TF-IDF = 0.095，高过 0.05 → 进入下一层 |
| ② | 查询词在语料库中完全不存在 | `drmtest` → BM25=0 → 拒绝 |
| ③ | 同分并列 + 零分不配当"第 2 名" | `format` 只在 1/35 个块中出现，其他块 BM25=0 → 末位 |
| ⑤ | TF-IDF 绝对零分块 | 查询与文档无任何字符重叠 |
| ⑥ | RRF 排名不够靠前 | 至少一个 ranker 把它排进前列 |

**关键词高亮 (`highlight_text`):**

```
输入: "报销流程", snippet = "报销流程说明..."
输出: "<mark>报销</mark><mark>流程</mark>说明..."
```

- 高亮标记使用 `<mark>` 标签
- CLI 输出使用 ANSI 转义序列（黄字）
- Web 界面使用 CSS 样式（黄底橙字）
- 匹配策略：中文单字 + 英文整词，与 tokenize 一致

**核心函数:**

| 函数 | 说明 |
|------|------|
| `tokenize(text)` | 中文单字 + 英文整词分词 |
| `load_index()` | 加载 chunks, vectorizer, tfidf_matrix, bm25_index, embeddings |
| `bm25_score(query, bm25_index)` | 计算查询对所有块的 BM25 得分 |
| `build_bm25_index(chunks)` | 构建 BM25 倒排索引（index_docs.py 调用） |
| `highlight_text(text, query)` | 返回含 `<mark>` 高亮的 HTML 字符串 |
| `hybrid_search(query, ...)` | 混合搜索主函数，返回结果列表 |
| `search(query, top_k)` | CLI 入口 |

### 3.4 index_docs.py — 索引构建模块

构建流程：

```
[1/4] 扫描文档目录 (docs/ + fetched_docs/)
[2/4] 分块 (段落分块 + 滑动窗口) + TF-IDF 向量化 + BM25 索引构建
[3/4] 向量嵌入生成（仅 --fetch 模式，sentence-transformers）
[4/4] 保存索引 (chunks, vectorizer, tfidf_matrix, bm25, embeddings)
[5/5] 统计输出
```

**分块策略:**

- 按段落分块（段落 = 连续非空行）
- 追踪每块在原文中的行号范围 (1-indexed)，便于 IDE 跳转
- 超长段落使用滑动窗口切分：`chunk_size=500, overlap=50`

### 3.5 server.py — Web 服务模块

`load_index()` 和 `do_search()` 均从 `search.py` 导入，消除代码重复。

**内置 HTML 搜索页面:**
- 路径 `/` 或 `/index.html`
- 搜索框 + 结果卡片 + 分页
- 结果支持关键词高亮 (`highlight` 字段优先于 `snippet`)
- 响应式布局，移动端友好

## 4. 数据流

### 4.1 搜索流程 (v1.4 混合搜索)

```
用户查询: "报销流程"
          │
          ▼
    tokenize("报销流程") → ["报", "销", "流", "程"]
          │
          ├──→ TF-IDF: vectorizer.transform → cosine_similarity → [N]
          │
          ├──→ BM25: 遍历 tokens → IDF × term_freq_score → [N]
          │
          └──→ Embedding: (可选) SentenceTransformer → cosine_similarity → [N]
          │
          ▼
    多层过滤:
          ├── max(TF-IDF) < 0.05? → 否
          ├── max(BM25) == 0?     → 否 (11 个块 > 0)
          ├── Dense ranking (同分同名次)
          ├── 零分块 → rank = n
          └── RRF 融合: Σ 1/(60+rank_i)
          │
          ▼
    排序 → 去重(按 source) → MIN_RELEVANCE 过滤 → Top-K
          │
          ▼
    高亮处理: highlight_text(snippet, query) → <mark>标签
          │
          ▼
    输出: [{title, source, snippet, highlight, score(0~1)}, ...]
```

## 6. 升级路线

### 当前: 混合搜索 (v1.4)

- TF-IDF char_wb + BM25 关键词 → RRF 融合
- 多层过滤防误报
- 关键词高亮（CLI ANSI + Web `<mark>`）
- 适用: < 200 篇文档

### 下一步: 向量嵌入 (v2.0)

- 在现有 BM25 + RRF 基础上增加稠密向量信号（三维 RRF）
- 模型: bge-small-zh-v1.5
- 解决"报销"搜不到"费用申请"的语义鸿沟

### 远期: RAG (v3.0)

- 搜索结果送给 LLM 生成自然语言答案

## 8. 企业微信接入设计

### 消息处理模块 (wechat.py)

| 函数 | 功能 |
|------|------|
| `calc_signature()` | SHA1 签名计算 |
| `decrypt(enc, key)` | AES-256-CBC 解密 |
| `encrypt(msg, key, corp_id)` | AES-256-CBC 加密 |
| `parse_encrypted_message(xml, key)` | 加密 XML → 明文消息 dict |
| `build_encrypted_reply(...)` | 构建加密回复 XML |
| `format_search_reply(query, results)` | 搜索结果 → 企微文本 |

### 回调验证流程

```
企业微信 → GET /wechat?echostr=xxx&msg_signature=xxx&timestamp=xxx&nonce=xxx
                │
                ├── 签名校验 (SHA1)
                ├── AES 解密 echostr
                └── 返回解密后的明文 echostr

企业微信 → POST /wechat <xml>
                │
                ├── 解析 XML（加密或明文）
                ├── 提取 Content 字段
                ├── 执行搜索
                └── 返回搜索结果 XML
```

回调 URL 配置: `https://your-domain.com/wechat`
