# 知识库搜索系统 — 需求文档

> 版本: v1.3 | 日期: 2026-06-07 | 状态: v1.3 已交付

---

## 1. 项目背景

团队约 20 人，日常依赖 Wiki 和 Markdown 文档进行工作。当前查找信息的方式是手动翻阅文档目录或 Wiki 页面，效率低下，尤其对新人而言。

**核心痛点:**

- 新人入职后不知道该查哪个文档、在哪里查
- 每个成员每天查文档 3-4 次，全团队每天约 60-80 次查询
- 每次手动翻找需耗时 2-5 分钟
- 本地文档和外部 Wiki 网页分散在不同系统，缺乏统一检索入口

## 2. 项目目标

构建一个内部知识库搜索系统，让团队成员通过关键词快速找到对应的文档内容。支持本地 Markdown 文档和外部网页文档的统一搜索。分阶段交付：

| 阶段 | 目标 | 状态 |
|------|------|------|
| MVP | CLI + Web API 搜索本地 Markdown 文档 | ✅ 已完成 |
| MVP 增强 | Web 搜索页面（浏览器输入框搜索） | ✅ 已完成 |
| v1.3 | 外部文档源支持（网页抓取 + 转换为本地索引） | ✅ 已完成 |
| Phase 2 | 接入企业微信，在企业微信内直接搜索 | 🔲 开发中 |
| Phase 3 | 混合搜索优化（向量 + 关键词）、新人 FAQ 专区 | 🔲 规划中 |

**成功指标:**

| 指标 | 目标值 | 当前值 |
|------|--------|--------|
| 搜索响应时间 | < 1 秒 | < 200ms |
| 搜索结果 Top-3 命中率 | > 80% | 待验证 |
| 新人上手时间 | < 5 分钟 | ✅ 开箱即用 |
| 系统可用性 | 99% (工作时间) | ✅ 单机无外部依赖 |
| 外部文档接入 | < 5 分钟配置 | ✅ 配置 EXTERNAL_SOURCES 即可 |

### 4.2 F9 外部文档抓取详细需求

**配置方式:**
- 在 `config.py` 的 `EXTERNAL_SOURCES` 列表中配置外部源
- 每个源包含: name（名称）、urls（URL 列表）、content_selector（正文选择器）、exclude_selectors（排除选择器）

**抓取行为:**
- `python index_docs.py --fetch` 触发抓取
- 每个 URL 抓取后缓存到 `fetched_docs/<source_name>/` 目录
- 抓取间隔 0.5 秒（防止被目标站点封禁）
- 失败不阻塞：单个 URL 失败不影响其他 URL 和索引构建

**HTML → Markdown 转换:**
- 自动提取页面标题（`<title>` 或 URL 路径）
- 移除脚本、样式、导航、页脚等非正文元素
- 保留标题层级、链接、列表、代码块、表格等结构
- 文档头注入来源 URL 和抓取时间戳

## 7. 约束与假设

| 约束 | 说明 |
|------|------|
| 文档格式 | 本地仅支持 Markdown (.md)；外部网页通过 HTML→MD 转换 |
| 文档位置 | 本地在 `docs/` 下，外部缓存在 `fetched_docs/` 下 |
| 索引更新 | 手动运行 `index_docs.py --fetch`，全量重建 |
| 部署环境 | 单机部署，不支持分布式 |
| 网页抓取 | 依赖目标站点可访问性，需互联网连接 |

## 8. 术语表

| 术语 | 说明 |
|------|------|
| 文本块 (Chunk) | 从文档中切分的检索单元，最大 500 字符 |
| TF-IDF | Term Frequency-Inverse Document Frequency，词频-逆文档频率 |
| char_wb | Character-level Word Boundary 分析器，在词边界内做字符 n-gram |
| 余弦相似度 | 衡量两个向量方向相似程度的指标 |
| 混合排序 | TF-IDF 相似度 + 关键词命中加权的综合排序算法 |
| 外部文档源 | 通过 URL 抓取的外部网页文档，经 HTML→MD 转换后纳入索引 |
| CSS 选择器 | 用于精准定位网页正文区域的 selector 表达式 |
