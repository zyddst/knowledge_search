团队约 20 人，日常依赖 Wiki 和 Markdown 文档进行工作。本地文档和外部网页分散在不同系统，缺乏统一检索入口。现有搜索系统需要更精准的关键词匹配和排序能力。

### 3.2 F2 混合搜索详细需求

**排序算法:**
- TF-IDF char_wb 余弦相似度 → 排名
- BM25 关键词匹配（词频饱和 + 长度归一化 + IDF）→ 排名
- Embedding 语义向量（可选，v2.0）→ 排名
- RRF（倒数排名融合）合成最终排序
- 相关度归一化到 0~1 区间

**分词策略:**
- 中文：单字切分（"开发环境" → "开","发","环","境"）
- 英文：整词保留（"VS Code" → "vs","code"）

**多层过滤:**
1. TF-IDF 绝对阈值：max raw TF-IDF < 0.05 → 无结果
2. BM25 零检查：max BM25 == 0 → 无结果（词汇不存在）
3. Dense ranking：同分同名次，零分块强制末位
4. 相关度阈值：RRF < 0.60 → 排除

### 3.3 F9 关键词高亮详细需求

**实现方式:**
- `highlight_text(text, query)` → 返回含 `<mark>` 标签的 HTML
- CLI 使用 ANSI 转义序列（`\033[33m` 黄字 + `\033[1m` 粗体）
- Web 使用 CSS 样式（`background: #fff3bf; color: #d48806`）

**匹配规则:**
- 与 `tokenize()` 一致：中文单字 + 英文整词
- 大小写不敏感（英文）
- 按关键词长度降序匹配（优先匹配长词）

**数据结构:**
```python
results.append({
    "title": chunk["title"],
    "source": chunk["source"],
    "snippet": snippet,          # 原始文本（前 200 字符）
    "highlight": highlight_text(snippet, query),  # 高亮版本
    "score": round(float(score), 4),
})
```

## 7. 约束与假设

| 约束 | 说明 |
|------|------|
| 文档格式 | 本地 .md；外部网页通过 HTML→MD 转换 |
| 索引更新 | 手动运行 `python index_docs.py --fetch`，全量重建 |
| 部署环境 | 单机，Python 3.8+ |
| 分词 | 纯正则，不依赖 jieba 等分词库 |
| 高亮 | 正则匹配，与分词策略一致（中文单字 + 英文整词） |

## 8. 术语表

| 术语 | 说明 |
|------|------|
| BM25 | Best Matching 25，概率检索模型，对词频饱和和文档长度归一化 |
| RRF | Reciprocal Rank Fusion，倒数排名融合，多排序信号的无权重融合方法 |
| Dense Ranking | 稠密排名：同分同名次（1,2,2,3...） |
| TF-IDF | 词频-逆文档频率 |
| RRF 归一化 | RRF 原始分 / 理论最大值 → 0~1 |
| `<mark>` | HTML5 高亮标签，用于关键词可视化标记 |
| char_wb | 基于字符的词边界分析器，保留词边界信息，适合中文 n-gram |
