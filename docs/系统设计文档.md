## 2. 技术选型

| 组件 | 方案 | 选型原因 |
|------|------|----------|
| 编程语言 | Python 3.11 | 团队熟悉，生态丰富 |
| 检索引擎 | scikit-learn TfidfVectorizer | 纯 Python，零原生依赖，Windows 无痛部署 |
| 分块策略 | 段落分块 + char_wb 分词 | 中文友好，n-gram 兼顾精确关键词和模糊匹配 |
| 排序算法 | TF-IDF 余弦相似度 + 关键词命中加权 | 简单有效，20 人规模足够 |
| 序列化 | pickle | Python 原生，读写速度快 |
| Web 框架 | http.server (标准库) | 零额外依赖，MVP 场景够用 |
| 前端 | 内联 HTML + 原生 JS | 单文件分发，零构建步骤 |

**为什么不用 ChromaDB / 向量嵌入？**

ChromaDB 依赖 ONNX Runtime，后者在部分 Windows 环境需额外安装 VC++ 运行时。MVP 阶段文档量小（<100 篇），TF-IDF 准确度足够，且部署复杂度为零。后续文档量增长后可升级到 embedding 方案。

## 3. 模块设计

### 3.1 项目结构

```
project/
├── docs/                    # 原始 Markdown 文档（用户维护）
│   ├── 新人指南.md
│   ├── API接口规范.md
│   ├── 需求文档.md
│   └── 系统设计文档.md
├── vector_db/               # 索引文件（自动生成）
│   ├── chunks.pkl           # 文本块列表 [{id, content, source, title}, ...]
│   ├── vectorizer.pkl       # TfidfVectorizer 实例
│   ├── tfidf_matrix.pkl     # TF-IDF 稀疏矩阵 (n_chunks × n_features)
│   └── meta.txt             # 元信息（文档数、块数、索引时间）
├── config.py                # 全局配置
├── index_docs.py            # 索引构建模块
├── search.py                # CLI 搜索模块
├── server.py                # Web 服务模块（API + 搜索页面）
├── requirements.txt         # 依赖声明
└── README.md                # 项目说明
```

### 3.2 config.py — 配置模块

```python
# 可调参数
DOCS_DIR = "./docs"       # 文档目录路径
VECTOR_DB_DIR = "./vector_db"  # 索引存储路径
CHUNK_SIZE = 500          # 文本块大小（字符数）
CHUNK_OVERLAP = 50        # 滑动窗口重叠量
TOP_K = 5                 # 返回结果数
```

### 3.3 index_docs.py — 索引构建模块

**职责:** 扫描文档目录 → 解析 Markdown → 分块 → TF-IDF 向量化 → 持久化

**核心函数:**

| 函数 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `find_markdown_files(dir)` | 目录路径 | `list[Path]` | 递归查找所有 `.md` 文件 |
| `chunk_markdown(path)` | 文件路径 | `list[dict]` | 按段落分块，保留源文件和标题元信息 |
| `build_index(dir)` | 文档目录 | — | 主流程：编排上述步骤，写入 `vector_db/` |

**分块策略:**

```
Markdown 文件
  │
  ├── 解析标题（第一个 # 作为文档标题）
  ├── 按空行分割为段落
  ├── 合并短段落直到接近 CHUNK_SIZE (500 chars)
  ├── 超长段落用滑动窗口切分（overlap=50 chars）
  │
  └── 每个块生成唯一 ID: md5(filepath:chunk_index)[:12]
```

**TF-IDF 参数:**

```python
TfidfVectorizer(
    analyzer="char_wb",     # 字符级 n-gram，词边界内
    ngram_range=(2, 4),     # 2-4 gram，兼顾短词和长词
    max_features=10000,     # 词汇表上限
)
```

选用 `char_wb` 分析器而非默认的词级分析器，原因：
- 中文文本没有天然的空格分词
- char_wb 在词边界内做字符 n-gram，对中文自然适配
- 无需引入 jieba 等分词库，保持依赖最小化

### 3.4 search.py — CLI 搜索模块

**职责:** 加载索引 → 接收查询 → TF-IDF 相似度 + 关键词加权 → 排序去重 → 输出

**混合排序算法:**

```
score(chunk, query) = TF-IDF_cosine(query, chunk)
                    + keyword_hits * 0.05    # 关键词在正文命中数
                    + title_hits * 0.2       # 关键词在标题命中数
```

**去重策略:** 同一源文件只保留得分最高的一个块（避免 4 个块都是同一个文档）。

**关键字提取:** 用正则 `[一-鿿]+|\w+` 提取中文词和英文词。

**复杂度:** O(N × F)，其中 N = 文档块数，F = TF-IDF 特征维度。100 篇文档场景下 < 100ms。

**输出格式:**

```
────────────────────────────────────────────────────────────
#1  [新人指南]  相关度: 0.14
    文件: docs\新人指南.md
    ──────────────────────────────────────────
    在 OA 系统 → 费用报销 → 新建报销单...

#2  [API接口规范]  相关度: 0.08
    文件: docs\API接口规范.md
    ──────────────────────────────────────────
    ...
────────────────────────────────────────────────────────────
```

### 3.5 server.py — Web 服务模块

**职责:** 提供 HTTP 搜索接口 + Web 搜索页面，供浏览器、企业微信、前端等外部系统调用。

**核心类与方法:**

| 方法 | 功能 |
|------|------|
| `_html(content)` | 返回 HTML 页面（UTF-8 编码） |
| `_json(data)` | 返回 JSON 数据（含 CORS 头，支持跨域访问） |
| `do_OPTIONS` | CORS 预检请求处理 |
| `do_GET` | 路由分发：`/` → 搜索页面，`/search` → JSON，`/health` → 健康检查 |
| `do_POST` | POST 路由：`/search` → JSON 搜索，`/wechat` → 企业微信回复 |
| `log_message` | 日志输出（含 Unicode 容错处理） |

**Web 搜索页面架构:**

搜索页面（`HOME_PAGE`）为内联 HTML，包含三大组件：

```
┌─────────────────────────────────────┐
│  📚 知识库搜索                       │  ← 页头 (.header)
│  输入关键词，快速检索团队文档          │
├─────────────────────────────────────┤
│  [________________🔍] [搜索]         │  ← 搜索框 (.search-box)
├─────────────────────────────────────┤
│  #1 文档标题         相关度: 0.14    │
│  📄 文件路径                        │  ← 结果列表 (.results)
│  内容摘要...                        │
├─────────────────────────────────────┤
│  #2 文档标题         相关度: 0.08    │
│  ...                               │
└─────────────────────────────────────┘
```

前端交互流程：
1. 用户在输入框输入关键词，点击搜索或按回车
2. JavaScript `fetch('/search?q=' + encodeURIComponent(q))` 调用后端 API
3. 后端返回 JSON，前端动态渲染结果卡片
4. 无结果时显示空状态提示，出错时显示错误信息

**接口定义:**

#### `GET /` 或 `GET /index.html`
返回 Web 搜索页面。

```
→ GET /
← 200 <!DOCTYPE html> ... (搜索页面 HTML)
```

#### `GET /health`
健康检查。

```
→ GET /health
← 200 {"status": "ok"}
```

#### `GET /search?q=<关键词>`
搜索知识库，返回 JSON 结果。

```
→ GET /search?q=报销流程
← 200 {
     "query": "报销流程",
     "results": [
       {
         "title": "新人指南",
         "source": "docs/新人指南.md",
         "snippet": "在 OA 系统 → 费用报销 → 新建报销单...",
         "score": 0.14
       }
     ]
   }
```

#### `POST /search`
同上，请求体 JSON。

```
→ POST /search  {"q": "开发环境搭建"}
← 200 { "query": "开发环境搭建", "results": [...] }
```

#### `POST /wechat`
企业微信消息回调（预留）。企业微信 Bot 发过来的消息体包含 keyword 字段。

```
→ POST /wechat  {"keyword": "新人入职"}
← 200 {"reply": "找到相关文档:\n【新人指南】docs/新人指南.md"}
```

#### `GET /wechat?echostr=xxx`
企业微信回调 URL 验证（预留）。

**Windows 编码兼容:**

Windows 终端默认 GBK 编码，中文输出有两处风险：
1. 启动横幅中的中文可能在 PowerShell 下无法显示
2. 请求 URL 含中文时 `log_message` 的 `print()` 抛出 `UnicodeEncodeError` 导致 500

已做三层防护：

| 层 | 位置 | 措施 |
|----|------|------|
| 1 | `main()` | `try/except` 安全包装 `sys.stdout` 为 UTF-8，失败则跳过 |
| 2 | `main()` | 所有启动 `print()` 加 `flush=True`，确保立即输出不被缓存 |
| 3 | `log_message` | 捕获 `UnicodeEncodeError`，降级输出 `<request>` 占位符 |

## 4. 数据流

### 4.1 索引构建流程

```
用户运行: python index_docs.py
          │
          ▼
    find_markdown_files("docs/")
          │
          ├── docs/新人指南.md
          ├── docs/API接口规范.md
          └── ...
          │
          ▼
    chunk_markdown() 对每个文件
          │
          ├── 解析标题（第一个 # 作为文档标题）
          ├── 按段落（空行）分割
          ├── 合并短段落 → 接近 CHUNK_SIZE
          ├── 超长段落滑动窗口切分 (overlap=50)
          └── 每个块附 {id, content, source, title}
          │
          ▼
    TfidfVectorizer.fit_transform(all_contents)
          │
          ├── 学习 char_wb n-gram 词汇表 (max_features=10000)
          └── 生成 TF-IDF 稀疏矩阵 (n_chunks × n_features)
          │
          ▼
    pickle.dump → vector_db/
          ├── chunks.pkl          # 块列表
          ├── vectorizer.pkl      # 向量化器
          ├── tfidf_matrix.pkl    # TF-IDF 矩阵
          └── meta.txt            # 元信息
```

### 4.2 搜索流程

```
用户入口:
  ├── 浏览器: http://localhost:8080 → 输入框 → JS fetch /search?q=xxx
  ├── CLI: python search.py "新人开发环境怎么搭"
  └── API: curl localhost:8080/search?q=xxx
          │
          ▼
    search.py / server.py → search(query)
          │
          ├── 加载 vector_db/ 中的索引（首次加载后常驻内存）
          │
          ▼
    vectorizer.transform("新人开发环境怎么搭")
          │
          └── 查询文本 → TF-IDF 稀疏向量 (1 × n_features)
          │
          ▼
    cosine_similarity(query_vec, tfidf_matrix).flatten()
          │
          └── 每个块的余弦相似度得分 [N]
          │
          ▼
    关键词提取 + 加权
          │
          ├── 正则提取: [一-鿿]+|\w+
          ├── 关键词集: {"新人", "开发环境", "怎么", "搭"}
          ├── 正文命中: keyword_hits * 0.05
          └── 标题命中: title_hits * 0.2
          │
          ▼
    综合排序 → 去重（按 source 文件路径） → Top-K
          │
          ▼
    输出: [{title, source, snippet, score}, ...]
```

### 4.3 搜索评分算法详解

```
最终得分 = TF-IDF 余弦相似度 + 关键词命中加权

其中:
├── TF-IDF 余弦相似度（主信号）
│   └── 范围 [0, 1]，衡量查询与文档块的语义相关性
│
├── 正文关键词命中 × 0.05（辅助信号）
│   └── 查询中的每个关键词在 chunk content 中出现则 +0.05
│   └── 权重较低，避免偶然命中过度影响排序
│
└── 标题关键词命中 × 0.2（强信号）
    └── 查询中的每个关键词在 chunk title 中出现则 +0.2
    └── 权重较高，标题命中通常表示高相关性
```

**设计考量:**
- 标题命中权重 (0.2) 是正文命中 (0.05) 的 4 倍，因为标题通常反映文档主题
- 权重叠加后不会超过 TF-IDF 相似度的影响，保持关键词匹配为辅助角色
- 当查询完全不在词汇表中时，关键词命中仍可提供有意义的结果

## 5. 部署说明

### 环境要求

- Python 3.8+
- scikit-learn >= 0.24

### 启动步骤

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 把 Markdown 文档放入 docs/ 目录

# 3. 构建索引
python index_docs.py

# 4. 搜索（CLI）
python search.py "关键词"

# 5. 启动 Web 服务
python server.py
# 浏览器访问 http://localhost:8080
```

### 文档更新流程

```bash
# 新增或修改 docs/ 下的文件后，重建索引即可
python index_docs.py
```

索引构建是全量重建，当前设计下每次运行会覆盖 `vector_db/` 目录。对于 < 100 篇文档，全量重建耗时在秒级，可接受。

### 故障排查

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `ModuleNotFoundError: sklearn` | 未安装依赖 | `pip install -r requirements.txt` |
| `[错误] 索引不存在` | 未构建索引 | 先运行 `python index_docs.py` |
| 中文乱码 | Windows 终端编码 | 使用 PowerShell 或 Windows Terminal；`server.py` 已做 UTF-8 兼容 |
| `[错误] 文档目录不存在` | docs/ 目录缺失 | 创建 `docs/` 目录并放入 .md 文件 |
| 搜索结果不相关 | 文档内容与查询不匹配 | 检查文档是否在 `docs/` 下；检查是否运行过 `index_docs.py` |
| 端口被占用 | 8080 端口已在使用 | `python server.py 9000` 换端口 |

## 7. 升级路线

### 当前: TF-IDF (v1.x MVP)

- 适用: < 100 篇文档
- 优点: 零依赖、部署简单
- 局限: 仅关键词匹配，无语义理解

### 下一步: 向量嵌入 (v2.0)

当文档量 > 100 篇或需要语义搜索时，升级到 embedding 方案：

- 模型: bge-large-zh 或 text2vec-base-chinese（中文优化）
- 向量库: ChromaDB 或 Milvus Lite
- 检索: 混合搜索（向量相似度 + BM25 关键词）
- 前置条件: 安装 VC++ 运行时 (onnxruntime 依赖)
- 迁移成本: 修改 `index_docs.py` 和 `search.py`，接口不变

### 远期: RAG + 答案生成 (v3.0)

- 搜索结果送给 LLM 生成自然语言答案
- 需要评估幻觉风险（内部文档场景下准确性 > 流畅性）
- 建议积累 3 个月搜索日志后，根据高频查询的满意度决定是否引入

## 8. 已知限制

| 限制 | 影响 | 缓解方案 |
|------|------|----------|
| 仅支持 .md 文件 | Word/PDF 文档需先转换 | 后续可加 `python-docx` / `pypdf` 支持 |
| 全量索引重建 | 文档多时耗时长 | 后续加增量索引（按文件哈希判断变更） |
| 无语义理解 | "报销"搜不到"费用申请" | v2.0 向量嵌入方案解决 |
| 单机部署 | 无法多人在不同机器访问 | server.py 可在局域网直接访问 |
| 无认证 | API 无鉴权 | 企业微信回调自带身份，内网场景可接受 |
| 无搜索结果高亮 | 页面摘要不标记匹配词 | 后续可在前端用 `<mark>` 标签高亮关键词 |
| 无搜索日志 | 无法分析搜索质量和热门查询 | 后续可加简单的 JSONL 日志 |
